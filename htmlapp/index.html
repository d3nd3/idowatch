IDO / VeryFit / Ryze based smartwatch activity downloader

This makes a bluetooth connection to IDO based smart watches and creates a TCX file from the activity data. This specifically doesn't mark the activity as sync'd to allow VeryFit/Ryze to continue operating

Requirements

  • Browser with bluetooth support : Currently only Chrome.
  • Activity hasn't be sync'd with VeryFit / Ryze app already

Limitations

  • Doesn't mark activities as sync'd, so you need to sync each activity with the app
  • Only supports running
  • Only supports a wide subset of the angelfit protocol; some devices may not support all commands
  • Doesn't support getting GPS path
  • May do unexpected things

Lets go!

Run from a local server (required for Bluetooth): In a terminal, from the project folder run cd htmlapp && python3 -m http.server 8000, then in Chrome open http://localhost:8000. Do not open the HTML file directly (file://) or Bluetooth will not work.

From new_setup.json: VeryFit enables notifications on 0x0AF7 and 0x0AF2, but only 0x0AF7 carries application payloads in this trace. The first host writes are 02 04 → 02 02 → 02 07 → 02 F0 → v3 activity; the early 03 30 55 01 00 packet is a watch status notification, not an app write and not live HR. No 03 27 / F0 01 reset command appears in the capture.


Send test notification: Mi Band style (0x05 0x01 + UTF-8) triggers call on many IDO watches. If it stops working, re-pair with VeryFit to restore call alert; avoid sending “Enable call alert first” (0x03 0x30) if already set via VeryFit—it can reset state.

Health/reminder (capture-backed):

User toggled in order: (1) Continuous HR on→off, (2) Stress on→off, (3) Drinking on→off, (4) Walk around on→off, (5) Menstrual on→off. Commands in healthdata_toggles.json: #1 0x45, #2 none (Stress; app-only or shares 0x45), #3 0x60, #4 0x47, #5 0x41+0x42. After 0x45 the app sends GET 0x08 (×4). hr_cont_enabled.json confirms 0x45 + GET 0x08. See TOOBUR.md.

Experimental (healthdata_toggles.json):

Test continuous heart rate: Live HR: -- If auto HR does not activate: check Func table (main heart_rate(5), Other2 heart_rate_monitor(3)); try «Alternative order», «v2 byte 2», or «v2 byte 3». Do not treat every 0x03 packet as HR: in new_setup.json, 03 30 55 01 00 is notice status, not 85 bpm. Also note that healthdata_toggles.json did not show VeryFit using legacy 0x25 when its app-side continuous HR / stress toggles changed; the only candidate there is a newer 0x45 health-config family.
Other (cmd 0xF0):

Alarms (SET 0x03 0x02)

Only if func table alarm_num > 0. Payload: alarm_id, status (0xAA=on, 0x55=off), type, hour, minute, repeat, snooze_min.

Set single alarm: Slot Type Repeat Snooze min
Full alarm sync: Build a list, then send all to the watch (one packet per alarm, 300 ms apart). 0 alarms in list

Sleep (goal + data / REM stages)

Set daily sleep target and request sleep data (watch may report stages: awake, light, deep, REM). Only if func table main_func sleep(1) is set.

Sleep goal (target duration):
Request sleep data: Watch may stream today’s or history sleep (stages: awake, light, deep, REM). Replies on 0x0AF7; may be multi-packet.

Sleep may be reported as Light / Deep segments (start time + duration). The event log shows each segment with human time, Unix timestamps, and hex bytes — search for that hex in veryfit_wireshark.json or BLE logs to find the same data in the capture.

Learn / probe

Send raw bytes from a capture (e.g. healthdata_toggles.json, new_setup.json) or probe HEALTH_DATA keys to see what the watch returns.

Send raw hex to 0x0AF6: Paste a line from Wireshark JSON btatt.value (e.g. 03:45:aa:09:00:12:00:55:3f:3c:00:50:00:00:00:1e). Spaces or colons allowed.

HEALTH_DATA (cmd 0x08) key probe: Request sync start with key (hex) Known: 04=today sleep, 06=history sleep, A0=live (some). Try 41, 42, 45, etc. to discover more.
